# 数据清洗计划

## 1. 总体判断

当前资料足以启动“院校查询 + 专业查询”的第一版数据层，但不足以直接完成可靠的“冲稳保推荐”数据层。

原因：

- 结构化表格只有院校名单和专业目录
- 真正关键的录取数据主要在 PDF 中
- 其中 `[历史类]2024~2025年录取统计.pdf` 有一定可提取性，但噪声较大
- `[物理类]2024~2025年录取统计.pdf` 当前抽样无法直接提取文本，后续很可能需要 OCR 或人工辅助

## 2. 清洗目标

第一阶段清洗目标不是一次做全，而是分成两批：

### 第一批：优先清洗，支撑基础查询页

1. `全国普通高等学校名单.xls`
2. `普通高等学校本科专业目录.xlsx`
3. `普通高等学校高等职业教育（专科）专业目录.xlsx`

输出目标：

- `schools.json`
- `majors.json`

### 第二批：重点攻关，决定推荐功能是否可落地

1. `[历史类]2024~2025年录取统计.pdf`
2. `[物理类]2024~2025年录取统计.pdf`

输出目标：

- `admissions.json`
- `rank_tables.json`

## 3. 分文件清洗策略

### 3.1 `全国普通高等学校名单.xls`

目标：

- 生成院校基础表

主要问题：

- 省份是通过分段标题行表示，不是逐行字段
- `所在地` 字段可能既有省又有市，需规范化
- 数据时间较旧，需要在文档中标注时效风险

建议清洗动作：

1. 跳过标题说明行
2. 识别“北京市（91所）”这一类省份分组行
3. 将当前分组省份回填到后续院校记录
4. 清洗空格和换行
5. 统一字段命名
6. 输出基础院校标签字段：
   - `school_name`
   - `province`
   - `city`
   - `ownership`
   - `school_level`
   - `notes`

风险：

- 缺少 `985/211/双一流/院校类型/官网` 等字段
- 这些字段需要后续补充源，不应凭经验猜测

### 3.2 `普通高等学校本科专业目录.xlsx`

目标：

- 生成本科专业清单

主要问题：

- 同一列里混有学科门类、专业类、专业三级结构
- 需要把层级关系拆出来

建议清洗动作：

1. 识别 2 位代码为学科门类
2. 识别 4 位代码为专业类
3. 识别 6 位或带字母代码为具体专业
4. 给每个专业补上所属门类和所属专业类
5. 输出统一字段：
   - `major_code`
   - `major_name`
   - `major_category`
   - `discipline_category`
   - `degree`

风险：

- 无选科要求、无学制、无专业简介

### 3.3 `普通高等学校高等职业教育（专科）专业目录.xlsx`

目标：

- 生成专科专业清单

主要问题：

- 存在目录行、专业大类行、专业类行、正式数据行混杂
- 还存在“新旧专业对照”和“增补专业”两套结构

建议清洗动作：

1. 分别处理两个 sheet
2. 只把正式专业记录作为主表
3. 将 `原专业代码`、`原专业名称`、`调整情况` 作为扩展字段保留
4. 将 `专业大类`、`专业类` 归并到主专业记录

风险：

- 与本科目录合并时要统一字段规范

### 3.4 `[历史类]2024~2025年录取统计.pdf`

目标：

- 尝试提取历史类录取数据与分数段表

主要问题：

- 文本可提取，但噪声大、乱码多、版式复杂
- 很可能需要按章节拆分处理：
  - 院校名称代号索引
  - 分数线及分数段
  - 招生计划或录取统计正文

建议清洗动作：

1. 先做 PDF 页面抽样和章节定位
2. 单独抽取目录索引页
3. 单独抽取分数段页
4. 判断正文表格是否可按列切分
5. 把 OCR/乱码风险高的页单独列入人工复核清单

风险：

- 可能无法在纯自动化下得到高质量 `admissions.json`
- 需要预留人工校对步骤

### 3.5 `[物理类]2024~2025年录取统计.pdf`

目标：

- 判断是否能自动提取

当前判断：

- 抽样前多页未提取到文本
- 高概率是扫描版或图像主导版 PDF

建议清洗动作：

1. 先做少量页面图像化抽样验证
2. 如确认无文本层，再决定是否引入 OCR
3. 若 OCR 成本过高，第一版可先只做历史类或只做查询功能

风险：

- 这是当前推荐功能的最大不确定项

## 4. 建议的数据标准化顺序

1. 建立项目目录：
   - `app/`
   - `scripts/`
   - `docs/`
2. 先写数据探查脚本：
   - `scripts/inspect_data.py`
3. 再写 Excel 转 JSON 脚本：
   - `scripts/convert_excel_to_json.py`
4. 再写校验脚本：
   - `scripts/validate_data.py`
5. 最后再处理 PDF 录取数据

## 5. 第一版 MVP 的现实方案

如果下一阶段希望尽快做出可运行网页，建议采用两步走：

### 方案 A：低风险起步

- 先做：
  - 院校查询
  - 专业查询
  - 收藏功能
  - 数据说明页
- 推荐页先使用“演示级规则 + 少量人工整理样例数据”

优点：

- 开发快
- 不被 PDF 清洗卡死

缺点：

- 推荐能力有限

### 方案 B：先攻克录取数据

- 先集中处理 `[历史类]2024~2025年录取统计.pdf`
- 争取做出一个单省单科类的真实推荐 MVP

优点：

- 更接近最终核心价值

缺点：

- 数据清洗风险高
- 工期更不确定

## 6. 当前缺失的关键数据

以下数据若能补充，将显著降低后续实现难度：

1. 结构化的近两年录取 Excel/CSV，而不是 PDF
2. 结构化的一分一段表
3. 结构化的 2026 招生计划表
4. 院校标签表：
   - `985`
   - `211`
   - `双一流`
   - 院校类型
   - 所在城市
5. 专业选科要求表

## 7. 下一步建议

建议下一步只做“数据标准化的第一小步”：

1. 先创建项目目录结构
2. 先把 3 个 Excel 清洗成基础 `schools.json` 和 `majors.json`
3. 暂时不承诺马上拿下完整 `admissions.json`
4. 单独对两个录取统计 PDF 做可提取性验证与试抽样

这样可以先把不确定性最大的部分隔离出来，避免前端开发建立在不稳定数据之上。
